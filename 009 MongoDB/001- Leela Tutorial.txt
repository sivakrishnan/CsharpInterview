2. Explain CAP Theorem Consistency, Availability and the Partition Tolerance  - Duration: 12 :45

It is very important to understand the limitations of NoSQL database. NoSQL can not provide consistency and high availability together. This was first expressed 
by Eric Brewer in CAP Theorem.

CAP theorem or Eric Brewers theorem states that we can only achieve at most two out of three guarantees for a database: Consistency, Availability and Partition tolerance

Consistency ---> means that all nodes in the network see the same data at the same time or the reader gets the most recently written data.
Availability ---> is a guarantee that every request receives a response about whether it was successfull or failed. However it does not guarantee that a read
request returns the most recent write. The more number of users a system can cater to better is the availability.
Partition Tolerance ---> is a guarantee that the system continues to operate despite arbitrary message loss or failure of part of the system. In other words,
even if there is a network outage, in the data center and some of the computers are unreachable, still the system continues to perform.

out of these three guarantees, no system can provide more than 2 guarantees. Since in the case of a distributed systems, the partitioning of the network is must,
the tradeoff is always between consistency and availability.


Consistency ---> All clients see the same view of data, even right after update or delete.
Availability ---> All clients ca n find a replica of data, even in case of partial node failures.
Partition Tolerance ---> The system continues to work as expected, even in presence of partial network failure

RDBMS ---> can provide only Consistency but not Partition Tolerance.
MongoDB, HBase and Redis can provide Consistency and Partition Tolerance.

CouchDB, Cassandra can provide only availability and Partition Tolerance but no consistency. Such database generally settle down for eventually consistency
and settle down after some time. Means after while the system will provide consistent data.

3. Different types of noSQL databases like Document, wide-column store, key-value   - Duration: 08 :51

The popular  Relational database - SQL server, Oracle, MySQL, MariaDB, postgreSQL

Document database ------------>
Document database store data in document similar to JSON objects. JSON stands for Javascript Object Notation. Each document contains pair of fields and values
and the values can have different data types like strings, numbers, arrays and even other objects. Document database is very popular because it can hold 
structures and unstructured both the kind of data very easily. If you have a need to build a very complicated data structure where you want to retrieve data very
quickly. Document database are usuall a good choice.

The popular document databases are MongoDB, CouchDB

Key Value database ------------>

These are simpler type of databases where each item contains a key and a value. A value can typically only be retrieved by referencing only its key.
So querying the key-value database is very simple. If your use case is simple where there is no need to store large amounts of data or you do not have to perform a very 
complex query. Key Value database is a good choice. 

The popular Key Value database are Redis and DynamoDB

wide column stores database ------------>

This is very similar to relational database because it stores data into tables and rows. however there is one aspect of the wide column store which is very
different from relational database is dynamic columns. wide-column stores provide a lot of flexibility over relational database. because each row is not required
to have the same columns. wide-column stores are great when you need to store a large amount of data and you can predict its query pattern.

The popular Key Value database are cassandra and HBase.

Graph database ------------>

 These are store data into nodes and edges. Nodes typically store information about people, places or various things which are typically stored into relational databases.
whereas edges store information about the relationship between those nodes. If you are storing people into graph databases, then the edges will contain information.
Graph databases are great when you want to traverse relationships to look for pattern like fraud detections.

The popular Key Value database are Neo4j  and JanusGraph

4. Advantages of using the Document Database - MongoDB    - Duration: 10:10

1) Intuitive Data Model---> Documents mapped to the object in your code is more natural to work with. Every data which you need to retrieve, you will keep them into a single document. There is no need to
have expensive joins or divide your data into multiple tables or multiple objects like relational database in case of document database. Data that is accessed together and is stored together so you
have to write less code and you can retrieve data quickly and efficiently.

2) Flexible Schema ---> The document database is dynamic and self describing. You can have any structured or unstructured data inside your documents. You can create a document which contains array and element
of the array can further contain different arrays. There is no set of rules how you can build your schema that provides a lot of flexibility of how you want to store a data. For a dynamic business like
ecommerce, of you want to add one particular attribute to one of your product, you just dont have to create a new column or new schema. In document database you can just go ahead and specify that one-time
unique data just fine inside your document. The flexibility of schema, an intuitive data model is possible because of the JSON documents.

Document database uses Javascript object Notation (JSON) to store all the data. JSON is a lightweight, language independent and human readable object.

when you are using JSON you can work with all the JSON document or JSON objects using a single query language giving user a consistent development experience.

3) Query Language itselft---> The language to query the data in mongodb is one of the most feature-rich query language. as mongodb is using the JSON document it is very easy to modify a single document or
multiple document altogether into different clusters. Dcoument database also supports very rich indexing so that we can retrieve data very efficiently from our document database.

4) Distributed Scalable Database ---> Scaling means that when you reach to a point where you need to scale up the relational database, you add more resources like memory, CPU or I/O. In the document database
you do not have to scale up vertically. You always do horizontal scaling what it means is that you can depend on the commodity hardware to host your document databases. Documents are independent units which
makes it easier to distribute them across multiple server while we preserve data locally. It is very easy to maintain your data into multiple commodity hardware where you have feature-rich replication and self
headline methodologies
